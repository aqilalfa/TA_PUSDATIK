# BAB 4.5: QUALITY GATES DAN GUARDRAILS UNTUK SISTEM SAFETY

## 4.5.1 Peran Guardrail pada Program Saat Ini

Pada program ini, guardrail tidak berfungsi sebagai pemblokir keras di depan model, tetapi sebagai instruksi tambahan dan evaluator pasca-generasi. Artinya, sistem menanamkan arahan kualitas ke dalam prompt, lalu menilai hasil jawaban setelah model selesai menghasilkan teks. Pendekatan ini sesuai dengan implementasi `build_quality_guardrail()` dan `build_answer_quality_report()`.

## 4.5.2 Guardrail Berbasis Konteks

### 4.5.2.1 Guardrail untuk pertanyaan tabel

Ketika query mengandung label tabel dan konteks memang memuat tabel tersebut, engine membangun instruksi tambahan agar model tetap membaca isi tabel secara literal. Di sini, tujuan utamanya bukan memvalidasi keamanan, melainkan mencegah jawaban yang terlalu bebas sehingga nilai baris atau kolom berubah.

### 4.5.2.2 Guardrail untuk pertanyaan umum

Untuk pertanyaan umum, guardrail menekankan agar jawaban tetap fokus pada istilah kunci yang memang muncul di konteks. Jika query meminta daftar atau rincian, instruksi tambahan meminta model menuliskan butir utama secara lengkap. Ini selaras dengan sifat dokumen SPBE yang sering memuat banyak subbagian dan item bernomor.

## 4.5.3 Quality Scoring Setelah Generasi

### 4.5.3.1 Skor kualitas jawaban

Fungsi `build_answer_quality_report()` menghitung beberapa sinyal utama, antara lain panjang jawaban, cakupan istilah inti, jumlah sitasi, kelengkapan daftar, dan keberadaan klaim yang menyatakan informasi tidak tersedia. Skor ini dipakai untuk menilai apakah jawaban cukup baik atau perlu ditinjau ulang.

### 4.5.3.2 Deteksi klaim yang tidak konsisten

Program menggunakan pola teks untuk mendeteksi klaim seperti “tidak ditemukan” atau “tidak tersedia”. Klaim tersebut tidak otomatis salah, tetapi akan dianggap bermasalah jika konteks sebenarnya masih mengandung istilah inti dari query. Dengan cara ini, sistem mencoba menekan jawaban yang menolak informasi padahal bukti dokumen sebenarnya ada.

## 4.5.4 Apa yang Tidak Dilakukan Program Saat Ini

Penting untuk dicatat bahwa sistem saat ini belum mengeksekusi blokir keras berbasis validator pre-generation di jalur chat utama. Tidak ada jalur yang secara eksplisit menghentikan prompt karena dianggap jailbreak atau toxic sebelum LLM dipanggil. Yang ada adalah scoring dan instruksi tambahan yang membentuk perilaku model, lalu hasilnya dianalisis setelah streaming selesai.

## 4.5.5 Implikasi pada Bab Penelitian

Karena sifatnya evaluatif, bagian ini sebaiknya dijelaskan sebagai quality assurance berbasis heuristik, bukan sebagai sistem keamanan generatif penuh. Dengan framing yang tepat, pembaca akan memahami bahwa program memprioritaskan grounding dan konsistensi jawaban, bukan sekadar menampilkan label guardrail yang bersifat teoritis.

## 4.5.6 Ringkasan

Subbab ini menegaskan bahwa quality gate pada program sekarang bekerja sebagai kombinasi guardrail tekstual, scoring jawaban, dan deteksi klaim tidak konsisten. Narasi yang paling akurat adalah bahwa program meningkatkan kualitas jawaban melalui evaluasi post-generation dan instruksi konteks, bukan melalui blokir keamanan keras di setiap tahap.

## 4.5.7 Audit Konsistensi Sitasi (Grounding Check)

Untuk memastikan jawaban AI benar-benar berpijak pada dokumen sumber (*grounding*), sistem menjalankan fungsi `validate_answer` setelah jawaban dihasilkan. Fungsi ini membandingkan klaim dalam jawaban dengan data di jendela konteks.

```python
def validate_answer(answer: str, context: str) -> Dict:
    """Validasi bahwa jawaban didasarkan pada konteks (Grounding)."""
    # 1. Periksa keberadaan sitasi [n]
    citations = re.findall(r"\[(\d+)\]", answer)
    if not citations:
        return {"is_valid": False, "warning": "Jawaban tidak memiliki sitasi"}

    # 2. Periksa halusinasi nomor Pasal/Ayat
    answer_pasals = set(re.findall(r"Pasal\s+(\d+)", answer, re.IGNORECASE))
    context_pasals = set(re.findall(r"Pasal\s+(\d+)", context, re.IGNORECASE))
    
    hallucinated_pasals = answer_pasals - context_pasals
    if hallucinated_pasals:
        return {
            "is_valid": True, 
            "confidence": "medium",
            "warning": f"Pasal {list(hallucinated_pasals)} tidak ada di sumber"
        }
```

**Analisis Teknik Implementasi:**
1.  **Pendeteksian Halusinasi Nomor**: Dalam domain hukum, kesalahan satu nomor pasal saja berakibat fatal. Dengan membandingkan set nomor pasal yang muncul di jawaban vs konteks, sistem dapat memberikan peringatan "medium confidence" kepada pengguna jika AI mencoba mengarang nomor pasal.
2.  **Audit Sitasi**: Sistem mewajibkan adanya penanda `[n]`. Jika AI memberikan jawaban panjang tanpa referensi, sistem akan menandainya sebagai tidak valid untuk menjaga akuntabilitas informasi.

## 4.5.8 Audit Hierarki SPBE (Domain-Specific Guardrail)

Karena dokumen Laporan Evaluasi SPBE memiliki hierarki yang kaku (Domain > Aspek > Indikator), LLM seringkali tertukar dalam menyebutkan level tersebut. Sistem mengatasi hal ini dengan audit hierarki otomatis.

```python
def _check_spbe_hierarchy_confusion(answer: str, result: Dict):
    """Mencegah tertukarnya istilah Aspek dan Indikator dalam jawaban."""
    for indicator_name in _SPBE_INDICATOR_NAMES:
        if indicator_name in answer_lower:
            idx = answer_lower.find(indicator_name)
            context_before = answer_lower[max(0, idx - 200):idx]
            # Deteksi jika indikator disebut sebagai 'aspek'
            if "aspek" in context_before and "indikator" not in context_before:
                result["warnings"].append(f"Indikator {indicator_name} tertukar sebagai Aspek")
```

**Analisis Teknik Implementasi:**
1.  **Semantic Context Awareness**: Audit ini tidak hanya mencari kata kunci, tapi melihat konteks 200 karakter sebelum istilah tersebut muncul. Jika kata "Aspek" muncul tepat sebelum nama sebuah Indikator, sistem mendeteksi adanya kebingungan hierarki.
2.  **Domain Guardrail**: Ini adalah contoh *logic-based guardrail* yang lebih efisien daripada menggunakan LLM lain untuk memvalidasi jawaban, karena bekerja secara deterministik pada aturan bisnis yang sudah pasti.
