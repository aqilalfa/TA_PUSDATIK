# BAB 4.6: AUTENTIKASI, OTORISASI, DAN PENGELOLAAN SESSI

## 4.6.1 Arsitektur Autentikasi

Komponen autentikasi pada program ini dibangun di atas JWT dan audit log, bukan pada sesi Redis yang berdiri sendiri. Saat pengguna login, sistem membuat access token dan refresh token. Refresh token disimpan sebagai cookie HttpOnly, sedangkan access token dikembalikan ke klien untuk dipakai pada request berikutnya. Pendekatan ini sesuai dengan implementasi `auth_routes.py`.

## 4.6.2 Login Lokal dan LDAP

### 4.6.2.1 Provider autentikasi

Sistem mendukung pemilihan provider autentikasi melalui konfigurasi. Jika LDAP aktif, pengguna dapat diautentikasi melalui direktori LDAP/Active Directory. Jika tidak, sistem dapat memakai sumber lokal. Pada saat autentikasi berhasil, peran pengguna diparsing dari role mapping dan dibawa ke dalam claim JWT.

### 4.6.2.2 Shadow user provisioning

Pada implementasi LDAP, pengguna yang lolos autentikasi dapat dipetakan ke user lokal agar sistem masih memiliki representasi internal untuk audit, percakapan, dan hak akses. Jadi, LDAP berfungsi sebagai sumber identitas, sedangkan basis data lokal tetap memegang jejak operasional aplikasi.

## 4.6.3 JWT, Refresh Cookie, dan Blacklist

### 4.6.3.1 Access token dan refresh token

Ketika login sukses, sistem membuat access token yang berisi subjek, nama pengguna, peran, departemen, sesi, dan provider autentikasi. Refresh token dibuat terpisah dan disimpan dalam cookie HttpOnly agar tidak mudah diakses oleh skrip sisi klien.

### 4.6.3.2 Refresh dan logout

Endpoint refresh mengambil token dari cookie, memverifikasinya, lalu mengeluarkan access token baru. Pada proses logout, access token yang sedang aktif dapat dimasukkan ke token blacklist berdasarkan `jti` sehingga token lama tidak dapat dipakai kembali sampai kedaluwarsa.

## 4.6.4 Pengelolaan Percakapan Bukan Sesi Autentikasi

Perlu dibedakan bahwa `session_id` pada chat bukan sesi autentikasi, melainkan pengenal percakapan. Di endpoint chat, session digunakan untuk menyimpan riwayat percakapan, mengaitkan pesan pengguna, dan menjaga konteks antarturn. Karena itu, nilai session pada chat lebih tepat dipahami sebagai conversation tracking daripada state keamanan.

## 4.6.5 Audit dan Jejak Aktivitas

Login, gagal login, refresh token, dan logout semuanya dicatat melalui audit logger. Ini memberi kemampuan pelacakan aktivitas pengguna dan mempermudah analisis keamanan bila terjadi anomali. Pada level aplikasi, mekanisme audit ini lebih relevan dibanding rate limiting yang belum menjadi bagian inti pembahasan autentikasi pada kode saat ini.

## 4.6.6 Ringkasan

Subbab ini sebaiknya dibaca sebagai penjelasan autentikasi berbasis JWT, refresh cookie, blacklist token, LDAP/local provider, serta pemetaan role ke dalam claim token. Narasi ini lebih sesuai dengan implementasi aktual daripada deskripsi stateless session berbasis Redis atau DDoS protection yang belum menjadi inti kode autentikasi program.

## 4.6.7 Implementasi Autentikasi LDAP (Directory Services)

Sistem menggunakan `LDAPAuthProvider` untuk memvalidasi pengguna terhadap server direktori organisasi. Hal ini memungkinkan sistem untuk menggunakan kredensial tunggal (*Single Sign-On*) dan melakukan provisi pengguna secara otomatis (*shadow user*).

```python
def authenticate(self, username: str, password: str, db: Session) -> User | None:
    """Autentikasi terhadap LDAP dan pemutakhiran shadow user lokal."""
    try:
        # 1. Bind ke server LDAP untuk verifikasi password
        directory_user = self.directory_client.authenticate(username, password)
    except LDAPUnavailableError:
        raise
        
    if not directory_user:
        return None

    # 2. Sinkronisasi role berdasarkan grup LDAP
    roles = map_directory_groups_to_roles(directory_user.get("groups", []))
    return self._upsert_shadow_user(db, directory_user, roles)
```

**Analisis Teknik Implementasi:**
1.  **Shadow User Provisioning**: Teknik `_upsert_shadow_user` memastikan bahwa meskipun autentikasi dilakukan secara eksternal (LDAP), sistem tetap memiliki record pengguna lokal untuk keperluan audit logging dan manajemen sesi tanpa menyimpan password pengguna.
2.  **Role Mapping**: Keamanan PBAC (*Policy Based Access Control*) dibangun dengan memetakan grup LDAP (misal: "Admin_IT") ke dalam role sistem (misal: "admin"). Hal ini memudahkan pengelolaan akses di level organisasi.

## 4.6.8 Manajemen Sesi Aman (JWT & HttpOnly Cookie)

Untuk menjaga keamanan sesi setelah login, sistem menerbitkan JSON Web Token (JWT). Token akses dikirim sebagai JSON, sedangkan token penyegar (*refresh token*) disimpan di dalam *HttpOnly Cookie*.

```python
@router.post("/login")
def login_for_access_token(response: Response, form_data: OAuth2PasswordRequestForm):
    """Proses login OAuth2 dan penerbitan token sesi."""
    # 1. Verifikasi pengguna
    user = authenticate_user(form_data.username, form_data.password, db)
    
    # 2. Terbitkan Access Token
    access_token = security.create_access_token(user.id)
    refresh_token = security.create_refresh_token(user.id)
    
    # 3. Simpan Refresh Token di HttpOnly Cookie (Aman dari XSS)
    response.set_cookie(
        key="refresh_token",
        value=refresh_token,
        httponly=True,
        secure=True,
        samesite="lax"
    )
    return {"access_token": access_token, "token_type": "bearer"}
```

**Analisis Teknik Implementasi:**
1.  **Mitigasi XSS**: Dengan menyimpan *refresh token* di dalam *HttpOnly Cookie*, skrip JavaScript jahat tidak dapat mencuri token tersebut. Ini adalah standar keamanan industri untuk aplikasi web modern.
2.  **Audit Logging**: Setiap upaya login, baik berhasil maupun gagal, dicatat secara otomatis ke dalam tabel audit. Hal ini krusial untuk memenuhi standar kepatuhan (*compliance*) keamanan informasi dalam evaluasi SPBE.
