# BAB 4.3: MODEL LANGUAGE MODEL DAN STRATEGI PROMPTING

## 4.3.1 Pemilihan dan Konfigurasi LLM

Sistem RAG menggunakan **Qwen 2.5 3B Instruct** sebagai language model utama. Pemilihan model ini didasarkan pada beberapa pertimbangan teknis:

### 4.3.1.1 Kriteria Seleksi Model

**Model Size dan Efisiensi**:
- Model 3 miliar parameter dipilih untuk balance antara capability dan resource efficiency
- Dapat dijalankan pada GPU consumer-grade (NVIDIA GTX 1650 4GB VRAM)
- Inference latency < 2 detik per query pada hardware LSPro

**Instruction-Following Capability**:
- Model "instruct" telah di-finetune dengan instruction-following techniques
- Lebih responsif terhadap system prompts dan role-playing directives
- Better at structured output generation

**Multilingual Support termasuk Indonesian**:
- Qwen 2.5 dilatih pada data multilingual termasuk Indonesian
- Memahami konteks dan nuansa bahasa Indonesia
- Support untuk Indonesian formal language yang digunakan dalam peraturan

**Quantization Compatibility**:
- Available dalam format GGUF Q4_K_M (4-bit quantization)
- Quantization mengurangi model size dari ~6GB menjadi ~2GB
- Minimal impact pada quality sambil meningkatkan inference speed

### 4.3.1.2 Model Configuration dalam Ollama

Model di-serve melalui **Ollama** dengan konfigurasi berikut:

```
Model: qwen2.5:3b-instruct-q4_K_S
Parameters:
  - temperature: 0.0 (untuk deterministic output, minimal hallucination)
  - top_p: 0.95 (nucleus sampling)
  - top_k: 40 (top-k filtering)
  - max_tokens: 1024 (max response length)
  - repeat_penalty: 1.1 (penalize token repetition)
  - context_window: 4096 tokens
```

**Penjelasan parameter**:
- **temperature: 0.0**: Deterministik tanpa randomness. Penting untuk legal domain di mana konsistensi dihargai
- **top_p: 0.95**: Nucleus sampling untuk diversity terkontrol, mencegah repetition
- **repeat_penalty**: Mengurangi probability token yang sudah muncul, improving response coherence
- **context_window**: 4096 tokens cukup untuk konteks RAG + query + sistem instruksi

## 4.3.2 Sistem Prompt Engineering

Sistem menggunakan **multiple specialized prompts** yang dipilih berdasarkan query classification (lihat subbab 4.2.5):

### 4.3.2.1 SYSTEM_PROMPT_LEGAL

Untuk queries tentang pasal, ayat, dan peraturan:

```
Anda adalah asisten hukum yang menjawab pertanyaan tentang pasal dan ayat 
dari dokumen peraturan Indonesia.

ATURAN WAJIB:
1. JANGAN PERNAH mengarang nomor Pasal atau Ayat. Jika konteks tidak memiliki 
   nomor ayat, JANGAN tulis nomor ayat apapun.

2. Kutip teks PERSIS seperti yang tertulis dalam dokumen untuk nomor Pasal, 
   Ayat, dan daftar huruf.

3. Sertakan referensi [n] di setiap kalimat atau poin yang bersumber dari dokumen.

4. Jika ada daftar (a, b, c, d...), tulis LENGKAP semua butir yang ada di konteks.

5. JANGAN generalisasi atau menambahkan interpretasi di luar teks dokumen.

6. Jika informasi tidak ada dalam konteks, katakan: 
   "Informasi tersebut tidak ditemukan dalam dokumen yang tersedia."

7. Gunakan bahasa Indonesia formal pemerintahan.

FORMAT JAWABAN:
- Sebutkan nomor Pasal/Ayat sumber di awal jawaban
- Kutip isi pasal/ayat dengan referensi [n]
- Jika ada daftar huruf, sajikan lengkap dengan referensi per butir
```

Prompt ini spesifik untuk domain hukum karena:
- Enforces exact citation (mencegah misquoting peraturan)
- Prohibits hallucination of legal references
- Requires comprehensive lists (critical untuk daftar kondisi/persyaratan)

### 4.3.2.2 SYSTEM_PROMPT_SPBE

Untuk queries tentang SPBE (Sistem Pemerintahan Berbasis Elektronik):

```
Anda adalah asisten hukum Indonesia yang membantu menjawab pertanyaan 
tentang regulasi SPBE berdasarkan dokumen yang diberikan.

ATURAN WAJIB:
1. HANYA jawab berdasarkan dokumen yang diberikan
2. JANGAN mengarang Pasal atau Ayat yang tidak ada dalam konteks
3. Sertakan referensi [1], [2], [3] di setiap kalimat yang mengandung informasi
4. Jika ada daftar (a, b, c, d...), tulis LENGKAP semua butir
5. Jika informasi terlihat parsial, jelaskan bagian yang tersedia di konteks
6. Jika pertanyaan menyebut "Tabel X" dan konteks memuat "Tabel X", 
   jawab berdasarkan isi/nilai tabel tersebut
7. DILARANG menyatakan "tidak ditemukan" untuk "Tabel X" jika label "Tabel X" 
   ada di konteks

PENGETAHUAN STRUKTUR EVALUASI SPBE:
Struktur penilaian tingkat kematangan SPBE memiliki hierarki:
- DOMAIN (4): Kebijakan Internal, Tata Kelola, Manajemen, Layanan SPBE
- ASPEK (8): Aspek-aspek spesifik di bawah setiap domain
- INDIKATOR (47): Ukuran spesifik di dalam setiap aspek

ATURAN: JANGAN mencampurkan level hierarki. Indikator BUKAN aspek. 
Aspek BUKAN domain.
```

Prompt ini menambahkan domain knowledge tentang struktur SPBE untuk memastikan:
- Correct hierarchical references
- Accurate indicator/aspect/domain classification
- Consistency dengan SPBE framework resmi

### 4.3.2.3 SYSTEM_PROMPT_TABLE

Untuk queries tentang tabel dalam dokumen:

```
Anda adalah asisten yang membaca tabel dari dokumen pemerintah Indonesia.

ATURAN WAJIB:
1. Baca nilai tabel PERSIS seperti yang tertulis. JANGAN paraphrase atau ubah angka.
2. Sertakan SEMUA baris dan kolom yang ada di konteks. JANGAN lewati baris.
3. JANGAN mengarang nilai yang tidak tertulis di tabel.
4. Sertakan referensi [n] di setiap baris atau kelompok baris tabel.
5. Jika tabel tidak ada di konteks, katakan: "Tabel tidak ditemukan dalam 
   dokumen yang tersedia."
6. Gunakan bahasa Indonesia formal.

FORMAT JAWABAN:
- Mulai dengan menyebut nama/judul tabel yang ditemukan
- Sajikan isi tabel dalam format yang mudah dibaca (baris per baris atau 
  markdown table)
- Sertakan sitasi [n] di tiap baris
```

Prompt untuk table queries sangat ketat karena:
- Tabel sering mengandung data numerik yang critical
- Paraphrase atau rounding dapat mengubah makna
- Kelengkapan daftar tabel penting untuk accuracy

## 4.3.3 Prompt Engineering Architecture

### 4.3.3.1 User Prompt Template

User prompt dibangun dinamis dengan tiga komponen:

```python
USER_PROMPT_TEMPLATE = """
[INSTRUKSI]:
Ikuti langkah-langkah berikut untuk menjawab pertanyaan dengan akurat 
dan berdasar konteks.

1. Identifikasi bagian-bagian konteks yang relevan dengan memberi tag.
2. Untuk setiap bagian yang diberi tag, rangkum detail utama 
   (tanpa menambah informasi baru).
3. Gabungkan rangkuman-rangkuman ini dalam chain-of-thought 
   untuk menjawab pertanyaan.
4. Berikan jawaban akhir yang ringkas tanpa menunjukkan cara/sumber 
   Anda menemukan jawaban tersebut dan tanpa komentar tambahan.
5. Apabila rangkuman yang diperoleh tidak memadai untuk menjawab 
   pertanyaan, sampaikan permintaan maaf secara ringkas.

[GUARDRAIL]:
{guardrail_instructions}

[KONTEKS]:
{context}

[PERTANYAAN]:
{query}

[JAWABAN]:
"""
```

Komponen:
- **[INSTRUKSI]**: Chain-of-thought prompting untuk systematic reasoning
- **[GUARDRAIL]**: Dynamic guardrails berdasarkan query dan context (lihat subbab 4.5)
- **[KONTEKS]**: Retrieved dan reranked documents
- **[PERTANYAAN]**: User query
- **[JAWABAN]**: Placeholder untuk LLM response

## 4.3.4 Query Reformulation dan Expansion

### 4.3.4.1 Internal Query Reformulation

Sebelum di-retrieve, query melalui reformulation untuk improved retrieval:

```python
def reformulate_query(query: str, conversation_history: List[Dict]) -> str:
    """
    Reformulate query dengan context dari conversation history
    untuk disambiguasi dan context enrichment.
    """
    
    # Extract previous context
    recent_context = conversation_history[-4:]  # Last 2 turns
    
    # Build reformulation prompt
    reformulation_prompt = f"""
Diberikan percakapan berikut dan pertanyaan baru, 
reformulasi pertanyaan untuk menjadi standalone dan jelas:

Percakapan sebelumnya:
{format_conversation(recent_context)}

Pertanyaan baru: {query}

Reformulasi pertanyaan sehingga lebih jelas dan standalone:
"""
    
    # Call LLM untuk reformulation
    reformulated = llm.invoke(reformulation_prompt)
    
    return reformulated
```

Reformulation membantu dengan:
- Anaphora resolution (e.g., "Itu apa?" → "Apa definisi SPBE?")
- Ellipsis restoration (e.g., "Berapa jumlahnya?" → "Berapa jumlah domain SPBE?")
- Context injection dari conversation history

## 4.3.5 Temperature Control untuk Domain Spesifik

Model temperature dikontrol berdasarkan query type untuk balance antara:
- **Accuracy** untuk legal/regulatory queries (temperature: 0.0-0.1)
- **Flexibility** untuk explanation queries (temperature: 0.3-0.5)

```python
def get_optimal_temperature(query_type: str) -> float:
    """
    Tentukan temperature optimal berdasarkan query type.
    
    Lower temperature = more deterministic, focused (untuk legal)
    Higher temperature = more creative, varied (untuk explanations)
    """
    
    temperature_map = {
        "pasal": 0.0,           # Peraturan harus exact
        "table": 0.0,           # Tabel data harus exact
        "indikator": 0.1,       # Indikator relatif fixed
        "spbe": 0.2,            # SPBE bisa slightly varied
        "general": 0.3,         # General bisa lebih flexible
        "explanation": 0.5,     # Penjelasan bisa lebih kreatif
    }
    
    return temperature_map.get(query_type, 0.2)
```
