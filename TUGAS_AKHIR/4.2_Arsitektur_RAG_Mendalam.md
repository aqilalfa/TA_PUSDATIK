# BAB 4.2: ARSITEKTUR SISTEM RETRIEVAL-AUGMENTED GENERATION (RAG)

## 4.2.1 Gambaran Umum Arsitektur

Arsitektur RAG pada program ini terdiri atas empat bagian utama, yaitu klasifikasi query, retrieval hibrida, perangkuman konteks, dan generasi jawaban melalui model bahasa lokal. Alur ini diimplementasikan di dalam `LangchainRAGEngine` dan diekspos ke endpoint chat agar backend dapat menyiapkan konteks yang sesuai sebelum jawaban di-stream ke antarmuka pengguna.

Secara operasional, sistem bekerja dengan urutan berikut:

```
User Query
  ↓
classify_query()
  ↓
retrieve_context(query, top_k, doc_id)
  ├─ query expansion
  ├─ doc_id filter resolution
  ├─ vector search ke Qdrant
  ├─ BM25 search
  ├─ table literal search / indicator literal search
  ├─ RRF fusion
  ├─ neighbor-context stitching
  └─ final reranking + context formatting
  ↓
stream_answer(query, context, history, model_name, query_type)
  ↓
SSE token stream ke frontend
```

Model arsitektur ini dipilih agar query yang bersifat tabel, pasal, atau indikator tidak diperlakukan sama dengan query umum. Dengan demikian, retrieval dapat menyesuaikan jenis bukti yang dicari, bukan hanya melakukan pencarian semantik yang seragam.

## 4.2.2 Klasifikasi Query dan Routing Retrieval

### 4.2.2.1 Tipe Query yang Didukung

Fungsi `classify_query()` mengelompokkan pertanyaan ke dalam empat kategori, yaitu `table`, `indikator`, `pasal`, dan `general`. Klasifikasi ini tidak hanya bersifat labelisasi, tetapi juga menentukan jalur retrieval yang berbeda. Query bertipe tabel dan indikator mendapat kandidat konteks yang lebih banyak karena pencarian literal pada struktur seperti tabel atau daftar indikator cenderung memerlukan cakupan hasil yang lebih luas.

### 4.2.2.2 Filter Berdasarkan Dokumen

Program mendukung pencarian yang dibatasi pada dokumen tertentu melalui parameter `doc_id`. Di level engine, nilai ini diterjemahkan menjadi filter Qdrant sehingga pencarian hanya dilakukan pada chunk yang berasal dari dokumen target. Pendekatan ini penting ketika pengguna sedang membaca satu dokumen dan ingin menanyakan isi bagian tertentu tanpa tercampur dengan dokumen lain.

## 4.2.3 Hybrid Retrieval yang Dipakai Program

### 4.2.3.1 Jalur Retrieval Semantik dan Kata Kunci

Pipeline retrieval saat ini menggabungkan beberapa jalur pencarian sekaligus: pencarian vektor, pencarian BM25, pencarian literal untuk tabel, dan pencarian literal untuk indikator. Hasil dari beberapa jalur ini kemudian digabung menggunakan reciprocal rank fusion sehingga konteks yang memiliki sinyal relevansi kuat dari lebih dari satu jalur akan memperoleh peringkat lebih tinggi.

### 4.2.3.2 Mengapa Jalur Literal Diperlukan

Dokumen SPBE dan dokumen audit sering memuat informasi yang sangat spesifik, seperti nomor tabel, indikator, atau penomoran pasal. Pada kasus seperti itu, pencarian semantik saja tidak selalu cukup. Karena itu, engine menambahkan literal search untuk tabel dan indikator agar istilah yang benar-benar muncul di dokumen dapat diangkat tanpa bergantung penuh pada kemiripan semantik.

## 4.2.4 Reranking dan Penggabungan Konteks

### 4.2.4.1 Reranking Heuristik

Setelah kandidat konteks digabung, engine melakukan perankingan ulang menggunakan komponen ranker internal. Dalam implementasi yang berjalan, proses ini lebih dekat ke heuristik pengurutan dan pemilihan top-k berdasarkan sinyal retrieval gabungan, bukan semata-mata pemanggilan satu model reranker eksternal yang berdiri sendiri.

### 4.2.4.2 Neighbor Context Stitching

Salah satu fitur penting pada arsitektur ini adalah pengayaan konteks dengan chunk tetangga. Ketika satu chunk relevan ditemukan, engine dapat mengambil konteks sebelum dan sesudahnya agar isi yang diberikan ke model bahasa tetap utuh. Teknik ini membantu saat potongan dokumen hanya menampilkan sebagian kalimat, daftar, atau tabel.

## 4.2.5 Debug Retrieval dan Penelusuran Tahap demi Tahap

Program menyediakan endpoint debug retrieval yang menampilkan hasil pencarian secara transparan. Endpoint ini berguna untuk melihat hasil klasifikasi query, filter dokumen, kandidat vector search, BM25, literal search, dan hasil akhir sebelum jawaban dibentuk. Selain itu, skrip `rag_trace.py` digunakan untuk menelusuri pipeline secara berurutan sehingga proses retrieval dapat diaudit tanpa menebak-nebak isi konteks yang dipakai.

## 4.2.6 Alur ke Lapisan Generasi

Konteks yang telah diformat dari retrieval diteruskan ke lapisan generasi melalui fungsi streaming. Lapisan ini tidak hanya menerima isi konteks, tetapi juga jenis query, riwayat percakapan, dan model yang sedang dipakai. Hal ini membuat generasi jawaban mengikuti karakter pertanyaan dan prompt yang sesuai, misalnya prompt tabel untuk pertanyaan tabel atau prompt hukum untuk pertanyaan pasal.

## 4.2.7 Ringkasan

Arsitektur RAG pada program ini bukan sekadar kombinasi dense search dan sparse search. Implementasinya sudah mencakup query routing, document scoping, literal retrieval untuk tabel dan indikator, RRF fusion, context stitching, serta debugability melalui endpoint dan skrip trace. Seluruh mekanisme tersebut membuat pipeline retrieval lebih sesuai dengan bentuk dokumen SPBE yang digunakan oleh sistem.

### 4.2.8 Implementasi Orkestrasi RAG Engine

Inti dari kecerdasan sistem ini terletak pada kelas `LangchainRAGEngine` yang mengatur alur data dari pertanyaan pengguna hingga menjadi jawaban yang terverifikasi. Berikut adalah implementasi fungsi `retrieve_context` yang menunjukkan alur kerja 6 tahap:

```python
def retrieve_context(
    self,
    query: str,
    top_k: int = 5,
    use_rag: bool = True,
    doc_id: Optional[str] = None,
) -> Dict[str, Any]:
    """Pipeline Retrieval Utama: Search -> Fusion -> Stitch -> Final Rank."""
    # 1. Klasifikasi dan Ekspansi Query
    query_type = classify_query(query)
    search_queries = expand_query(query)
    
    # 2. Pencarian Paralel (Vector + BM25 + Literal)
    qdrant_filter = self._build_qdrant_filter(doc_id)
    v_docs = []
    for sq in search_queries:
        v_docs.extend(self.retriever.vector_search(sq, k, qdrant_filter))
        
    b_docs = self.retriever.bm25_search(query, k * 2, self._bm25_docs, doc_id)
    l_docs = self.retriever.table_literal_search(query, self.collection_name, doc_id)
    i_docs = self.retriever.indicator_literal_search(query, self.collection_name, doc_id)

    # 3. Hybrid Fusion menggunakan Reciprocal Rank Fusion (RRF)
    candidates = self.ranker.rrf_fusion([v_docs, b_docs, l_docs, i_docs], max_candidates=25)
    
    # 4. Context Stitching (Penggabungan chunk tetangga)
    expanded_docs = self.stitcher.expand_docs_with_neighbor_context(candidates, self.collection_name)
    
    # 5. Reranking Akhir dan Seleksi Top-K
    final_docs = self.ranker.rerank(query, expanded_docs, k)

    # 6. Formatting Konteks untuk LLM
    context = self._format_context(final_docs)
    return {"context": context, "sources": self._build_sources_list(final_docs)}
```

**Analisis Teknik Implementasi:**
1.  **Ekspansi Query**: Program tidak langsung mencari menggunakan query asli, melainkan melakukan ekspansi melalui `expand_query()`. Hal ini krusial dalam domain hukum untuk menangani sinonim atau singkatan (misalnya "SPBE" menjadi "Sistem Pemerintahan Berbasis Elektronik").
2.  **Scoping Dokumen**: Filter `qdrant_filter` memastikan bahwa sistem mematuhi batasan pencarian jika pengguna hanya ingin bertanya pada satu dokumen tertentu, mencegah terjadinya kontaminasi informasi dari dokumen lain.
3.  **Context Stitching**: Teknik `expand_docs_with_neighbor_context` memecahkan masalah chunking yang terlalu kaku. Jika sebuah pasal terpotong di tengah, sistem secara otomatis mengambil potongan sebelum dan sesudahnya agar LLM menerima informasi yang utuh.
