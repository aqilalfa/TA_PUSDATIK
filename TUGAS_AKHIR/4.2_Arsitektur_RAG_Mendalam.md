# BAB 4.2: ARSITEKTUR SISTEM RETRIEVAL-AUGMENTED GENERATION (RAG)

## 4.2.1 Overview Arsitektur RAG

Sistem RAG yang dikembangkan dalam penelitian ini terdiri atas tiga komponen utama yang bekerja secara sinergis: *information retrieval pipeline*, *context processing*, dan *LLM generation with safety constraints*. Desain arsitektur mengikuti prinsip *separation of concerns* untuk memastikan setiap komponen dapat dikembangkan, diuji, dan dioptimalkan secara independen sambil tetap terintegrasi dalam satu sistem yang kohesif.

Arsitektur RAG secara keseluruhan dapat digambarkan sebagai berikut:

```
User Query
    ↓
[1. Input Validation & Query Classification]
    ↓
[2. Query Expansion & Normalization]
    ↓
[3. Hybrid Retrieval] ← 
    ├─ Dense Retrieval (Semantic Search via Embeddings)
    ├─ Sparse Retrieval (BM25 Full-Text Search)
    └─ Ensemble Combination (50-50 weight)
    ↓
[4. Retrieved Contexts]
    ↓
[5. Reranking Layer] (Cross-Encoder Ranking)
    ↓
[6. Top-K Context Selection] (typically top 3-5)
    ↓
[7. Context Stitching & Formatting]
    ↓
[8. Prompt Engineering & Context Injection]
    ↓
[9. LLM Generation] (Qwen 2.5 7B)
    ↓
[10. Output Validation & Quality Gates]
    ↓
[11. Response Formatting]
    ↓
User Response + Citations + Images
```

Setiap tahapan memiliki tujuan spesifik dan mekanisme error handling yang terukur. Ketika salah satu tahapan gagal, sistem mengembalikan error response yang informatif kepada pengguna.

## 4.2.2 Komponen Embedding dan Vector Store

### 4.2.2.1 Sentence Embedding Model

Sistem menggunakan model embedding **Indo-Sentence-BERT** (indo-sentence-bert-base) yang dilatih khusus untuk bahasa Indonesia. Model ini dipilih karena:

1. **Optimized untuk Bahasa Indonesia**: Dilatih pada corpus besar bahasa Indonesia sehingga memahami semantik konteks lokal
2. **Ukuran Model Kecil**: ~135 juta parameter yang dapat di-cache dan dimuat dengan cepat
3. **High-Quality Embeddings**: 768-dimensional dense vectors yang robust untuk similarity search
4. **Normalized Output**: Output embedding sudah dinormalisasi sehingga cocok untuk cosine similarity

Implementasi embedding loader dalam sistem ini menggunakan custom wrapper `SBERTDirectEmbeddings` yang mewarisi dari `LangChain's Embeddings` base class:

```python
class SBERTDirectEmbeddings(Embeddings):
    """Direct SentenceTransformer wrapper yang bypass masalah tokenizer parallelism di Windows."""
    
    def __init__(self, model_name: str, cache_folder: str, device: str):
        import os
        os.environ["TOKENIZERS_PARALLELISM"] = "false"  # Cegah hang di Windows
        self.model = SentenceTransformer(
            model_name, 
            cache_folder=cache_folder, 
            device=device
        )
        
    def embed_documents(self, texts: List[str]) -> List[List[float]]:
        """Embed multiple documents (typically during indexing)."""
        return self.model.encode(
            texts, 
            normalize_embeddings=True
        ).tolist()
        
    def embed_query(self, text: str) -> List[float]:
        """Embed single query (during retrieval)."""
        return self.model.encode(
            [text], 
            normalize_embeddings=True
        )[0].tolist()
```

### 4.2.2.2 Qdrant Vector Database

Vector store menggunakan **Qdrant**, sebuah vector database yang dirancang untuk production use dengan fitur:
- **Fast Similarity Search**: Indexed search di atas 1 juta vectors dengan latency < 50ms
- **Payload Support**: Setiap vector dapat menyimpan metadata tambahan (document ID, page number, etc.)
- **Persistence**: Data stored di disk untuk durability
- **Scalability**: Support sharding untuk scale horizontal

Konfigurasi Qdrant dalam sistem:
- **Collection Name**: "spbe_rag_collection"
- **Vector Dimension**: 768 (sesuai dengan indo-sentence-bert)
- **Distance Metric**: Cosine similarity (standard untuk normalized embeddings)
- **Payload Fields**: 
  - `doc_id`: Reference ke document yang disimpan di SQLite
  - `chunk_index`: Urutan chunk dalam dokumen (untuk konteks retrieval)
  - `document_title`: Judul dokumen untuk display
  - `document_type`: Jenis dokumen (Peraturan, Laporan, Pedoman, etc.)

Proses indexing ke Qdrant:
1. Dokumen dipecah menjadi chunks (lihat subbab 4.4)
2. Setiap chunk di-embed menggunakan indo-sentence-bert
3. Embedded chunks disimpan ke Qdrant dengan metadata
4. Native Qdrant indexing otomatis mengoptimalkan vector search

### 4.2.2.3 Hybrid Retrieval: Dense + Sparse

Sistem retrieval menggunakan pendekatan **hybrid** yang menggabungkan:

**Dense Retrieval (Semantic-based)**:
- Menggunakan vector embeddings untuk menemukan dokumen dengan makna semantik serupa
- Query di-embed menggunakan model yang sama
- Similarity search menggunakan cosine distance
- Advantage: Robust terhadap paraphrase dan variasi bahasa
- Disadvantage: Sulit untuk exact phrase matching dan specific numbers

**Sparse Retrieval (Keyword-based)**:
- Menggunakan **BM25** (Best Matching 25), algoritma probabilistik untuk information retrieval
- Melakukan exact term matching dengan weighting berdasarkan term frequency dan document frequency
- Advantage: Sangat efektif untuk exact phrase dan entity matching (e.g., "Pasal 5", "Tabel 3")
- Disadvantage: Tidak memahami semantik, rentan terhadap synonym

Kombinasi hybrid ini dalam sistem:
```python
class HybridRetriever(BaseRetriever):
    def __init__(
        self,
        dense_retriever,      # MultiVectorRetriever dari Qdrant
        sparse_retriever,     # BM25Retriever
        dense_weight: float = 0.5,
        sparse_weight: float = 0.5
    ):
        self.dense_retriever = dense_retriever
        self.sparse_retriever = sparse_retriever
        self.dense_weight = dense_weight
```

## 4.2.3 Reranking Layer

Setelah hybrid retrieval mengembalikan dokumen-dokumen kandidat, tahapan selanjutnya adalah **reranking** menggunakan cross-encoder untuk memilih dokumen yang paling relevan.

### 4.2.3.1 Cross-Encoder Model

Sistem menggunakan model cross-encoder **BAAI/bge-reranker-v2-m3** yang:
- Merupakan specialized model untuk reranking tasks
- Input: (query, document) pairs
- Output: Relevance score antara 0-1
- Dilatih dengan supervised signal untuk menentukan relevansi pasangan query-document

Keunggulan cross-encoder dibanding bi-encoder:
- **Higher Accuracy**: Cross-encoder lebih akurat dalam menentukan relevance
- **Flexible Input Length**: Dapat menangani query + document yang lebih panjang
- **Better for Ranking**: Dirancang khusus untuk ranking tasks

### 4.2.3.2 Reranking Process

Tahapan reranking:
```python
class RAGRanker:
    def __init__(self, model_name: str = "BAAI/bge-reranker-v2-m3"):
        self.cross_encoder = CrossEncoder(model_name)
    
    def rerank(
        self, 
        query: str, 
        documents: List[Document],
        top_k: int = 3
    ) -> List[Document]:
        """Rerank documents berdasarkan relevance score."""
        
        # Prepare input pairs
        pairs = [(query, doc.page_content) for doc in documents]
        
        # Score pairs menggunakan cross-encoder
        scores = self.cross_encoder.predict(pairs)
        
        # Rank berdasarkan score
        ranked_indices = np.argsort(-scores)  # descending order
        
        # Return top-k
        return [documents[i] for i in ranked_indices[:top_k]]
```

Proses ini memastikan bahwa konteks yang dikirimkan ke LLM adalah yang paling relevan dengan query, meningkatkan kemungkinan LLM menghasilkan jawaban yang akurat.

## 4.2.4 Context Stitching dan Formatting

Setelah reranking, dokumen-dokumen terpilih harus diformat dan "dijahit" (stitched) menjadi konteks yang kohesif yang siap dikirim ke LLM.

### 4.2.4.1 Context Preparation

Tahapan context stitching:

1. **Document Deduplication**: Pastikan tidak ada dokumen duplikat dalam hasil retrieval
2. **Chunk Assembly**: Jika dokumen terdiri atas multiple chunks, assembly ulang menjadi bentuk yang readable
3. **Metadata Inclusion**: Sertakan metadata seperti document title, type, page number
4. **Text Formatting**: Format teks dengan struktur yang jelas menggunakan markup

```python
class ContextStitcher:
    def stitch(
        self,
        documents: List[Document],
        max_context_length: int = 3000
    ) -> str:
        """Prepare and format retrieved documents sebagai context."""
        
        context_parts = []
        current_length = 0
        
        for i, doc in enumerate(documents):
            # Extract content dan metadata
            content = doc.page_content
            doc_id = doc.metadata.get('doc_id', 'Unknown')
            title = doc.metadata.get('title', 'Untitled')
            doc_type = doc.metadata.get('type', 'Document')
            
            # Format dengan markup
            formatted = f"""
[Dokumen {i+1}]
Judul: {title}
Tipe: {doc_type}
ID: {doc_id}
Konten:
{content}
"""
            
            # Check length constraint
            if current_length + len(formatted) > max_context_length:
                break
            
            context_parts.append(formatted)
            current_length += len(formatted)
        
        # Combine semua parts
        return "\n---\n".join(context_parts)
```

### 4.2.4.2 Chunk-Level Retrieval Continuity

Sistem juga mempertahankan konteks antar-chunks untuk memastikan kontinuitas bacaan:

```python
def preserve_chunk_context(
    chunks: List[Document],
    retrieve_n_neighbors: int = 1
) -> List[Document]:
    """
    Jika chunk [15] diretrieve, include juga [14] dan [16]
    untuk mempertahankan reading continuity.
    """
    
    expanded_chunks = []
    
    for chunk in chunks:
        chunk_idx = chunk.metadata.get('chunk_index', 0)
        doc_id = chunk.metadata.get('doc_id')
        
        # Add neighbors (jika ada)
        for neighbor_offset in [-1, 0, 1]:
            neighbor_idx = chunk_idx + neighbor_offset
            if neighbor_idx >= 0:
                neighbor = fetch_chunk(doc_id, neighbor_idx)
                if neighbor and neighbor not in expanded_chunks:
                    expanded_chunks.append(neighbor)
    
    # Sort by original order
    expanded_chunks.sort(
        key=lambda x: (x.metadata['doc_id'], x.metadata['chunk_index'])
    )
    
    return expanded_chunks
```

## 4.2.5 Query Classification dan Expansion

Sebelum retrieval, sistem melakukan **query classification** dan **query expansion** untuk meningkatkan kualitas hasil retrieval.

### 4.2.5.1 Query Type Classification

Sistem mengklasifikasikan query ke dalam empat kategori:

```python
def classify_query(query: str) -> str:
    """
    Klasifikasikan query untuk routing ke prompt templates yang berbeda.
    
    Returns: 'table', 'pasal', 'indikator', atau 'general'
    """
    
    q = query.lower()
    
    if re.search(r'\b(tabel|table)\b', q):
        return "table"        # Q: "Apa isi Tabel 5?"
    
    if re.search(r'\b(indikator|id[-\s]*\d+)\b', q):
        return "indikator"    # Q: "Jelaskan Indikator 23"
    
    if re.search(r'\b(pasal|ayat|perpres|pp\s*\d+)\b', q):
        return "pasal"        # Q: "Apa isi Pasal 7 Ayat 3?"
    
    return "general"          # Q: "Bagaimana proses sertifikasi?"
```

Klasifikasi ini penting karena:
- **Table Queries** memerlukan ekstraksi data dari struktur tabel
- **Pasal/Ayat Queries** memerlukan exact citation dari peraturan
- **Indikator Queries** memerlukan referensi ke structured indicators
- **General Queries** dapat dijawab dengan paraphrase dan konteks umum

### 4.2.5.2 Query Expansion

Query expansion menggunakan teknik synonym expansion dan semantic reformulation:

```python
def expand_query(original_query: str) -> List[str]:
    """
    Expand query dengan synonyms dan reformulations untuk meningkatkan recall.
    """
    
    expanded = [original_query]
    
    # Indonesian-specific synonyms
    synonyms = {
        'jelaskan': ['apa', 'bagaimana', 'definisi dari'],
        'proses': ['tahap', 'langkah', 'prosedur'],
        'persyaratan': ['kondisi', 'kriteria', 'requirements'],
        'sertifikasi': ['sertifikat', 'certified'],
    }
    
    # Apply synonym replacement
    for original_word, synonym_list in synonyms.items():
        for synonym in synonym_list:
            reformulated = original_query.replace(
                original_word, 
                synonym
            )
            expanded.append(reformulated)
    
    # Semantic reformulation untuk common patterns
    if 'apa itu' in original_query.lower():
        # "Apa itu RAG?" → "Jelaskan RAG" / "Definisi RAG"
        term = original_query.replace('Apa itu', '').strip()
        expanded.append(f'Jelaskan {term}')
        expanded.append(f'Definisi {term}')
    
    return expanded[:5]  # Return top-5 variations
```

Expanded queries kemudian di-retrieve secara paralel dan hasil digabung dengan weighting terhadap original query.

## 4.2.6 Stream Processing Architecture

Untuk mendukung response latency yang rendah dan user experience yang baik, sistem mengimplementasikan **streaming response**:

```python
async def stream_answer(
    query: str,
    context: str,
    session_id: str,
    token_callback = None
) -> AsyncIterator[str]:
    """
    Stream jawaban token-by-token dari LLM untuk perceived responsiveness.
    """
    
    prompt = build_rag_prompt(query, context)
    
    # Stream dari Ollama
    async with AsyncClient() as client:
        async for chunk in client.chat(
            model='qwen2.5:3b-instruct-q4_K_S',
            messages=[...],
            stream=True
        ):
            token = chunk['message']['content']
            
            # Yield token untuk browser rendering
            yield token
            
            # Optional callback (e.g., untuk logging, token counting)
            if token_callback:
                await token_callback(token)
```

Streaming architecture ini memberikan:
- **Lower Perceived Latency**: User melihat jawaban mulai muncul segera, bukan menunggu complete response
- **Better UX**: Memberikan sense of progress
- **Token-Level Control**: Memungkinkan early stopping jika token tidak relevan

## 4.2.7 Error Handling dan Fallback Mechanisms

Sistem memiliki multi-level error handling:

```python
async def safe_rag_pipeline(query: str, session_id: str):
    """
    RAG pipeline dengan comprehensive error handling dan fallbacks.
    """
    
    try:
        # Level 1: Query Validation
        if not query or len(query) < 2:
            return {"error": "Query terlalu pendek"}
        
        # Level 2: Retrieval
        try:
            retrieved = await hybrid_retrieve(query, k=5)
            if not retrieved:
                # Fallback: Retrieve with relaxed filters
                retrieved = await hybrid_retrieve(query, k=10, similarity_threshold=0.3)
        except Exception as e:
            logger.error(f"Retrieval failed: {e}")
            return {"error": "Gagal mengambil konteks. Coba ulang."}
        
        # Level 3: Reranking
        try:
            reranked = ranker.rerank(query, retrieved, top_k=3)
        except Exception as e:
            logger.warning(f"Reranking failed: {e}, using unreranked results")
            reranked = retrieved[:3]
        
        # Level 4: Generation
        try:
            context = stitcher.stitch(reranked)
            answer = await generate_answer(query, context)
        except Exception as e:
            logger.error(f"Generation failed: {e}")
            return {"error": "Gagal menghasilkan jawaban"}
        
        # Level 5: Output Validation
        try:
            validated = await validate_output(answer, context)
            if not validated:
                return {"error": "Jawaban tidak memenuhi quality threshold"}
        except Exception as e:
            logger.warning(f"Validation failed: {e}, returning unvalidated answer")
        
        return {"answer": answer, "sources": [...]}
        
    except Exception as e:
        logger.critical(f"Unexpected error in RAG pipeline: {e}")
        return {"error": "Kesalahan sistem. Hubungi administrator."}
```

Mekanisme fallback ini memastikan sistem tetap berfungsi bahkan ketika beberapa komponen mengalami degradasi.
